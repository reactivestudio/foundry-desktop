# Дизайн-система foundry-desktop

Здесь лежат **вещи**: значения, части, знаки, витрина. Слова — законы,
обоснования, «почему так» — в каноне [docs/design](../docs/design/README.md),
15 глав прозы. Граница простая: канон объясняет, система показывает и отдаёт.
Расходятся — прав канон, но чинится это правкой `tokens.json`, а не спором.

## Три входа

| Хочу | Иду |
|---|---|
| **посмотреть, что есть** | [index.html](index.html) — витрина, открывается двойным щелчком |
| **добавить своё из сессии** | [parts/README.md](parts/README.md) — контракт части и обряд приёмки |
| **поменять значение** | [tokens/tokens.json](tokens/tokens.json) — и `python3 design/build.py` |

## Устройство

```
design/
  index.html            витрина — СОБИРАЕТСЯ скриптом, руками не править
  build.py              tokens.json → tokens.css + Tokens.swift + таблицы канона + витрина
  lint.py               проверки; он же на CI
  tokens/
    tokens.json         ★ единственный источник значений
    tokens.css          ← генерится, его линкует каждая часть
    Tokens.swift        ← генерится, его ест приложение
  parts/
    README.md           контракт части
    specimen.css        нейтральная сцена для образца
    1-foundation/       значения в лицо: палитра, поверхности, шкалы
    2-marks/            знаки: иконка, вордмарк, орб, лоадер, DMG
    3-elements/         кнопка, поле, чип, бейдж, точка статуса
    4-blocks/           строка инбокса, карточка канбана, строка диффа, KPI
    5-layers/           окно, панель, поповер, модалка, тост, завеса
    6-views/            целые экраны
    7-behaviour/        пустое, скелет, ошибка, прогресс, undo, Reduce Motion
  candidates/           приёмная: приехало из сессии, не разобрано, НЕ линтуется
  rejected/             кладбище: отвергнутое + причина отказа
```

## Почему так, а не иначе

**Один источник значений — потому что четыре уже проверены на нас.** Канон
объявил `13-tokens.md` единственным источником и получил четыре расходящихся
словаря: свой `:root` в каждом макете, у онбординга — другая шкала почти-чёрного,
у иконки — вообще хардкод и параллельная «webphone-палитра». Декларация не
работает; работает генерация. Теперь `tokens.css`, `Tokens.swift` и таблицы
значений в каноне **производятся** из `tokens.json` — разойтись им физически
нечем, а `lint.py` ловит попытку поправить сгенерированное руками.

**Витрина — сборка, а не файл.** Один большой канвас со всеми категориями
нужен, чтобы видеть систему целиком; отдельные части нужны, чтобы их можно было
делать в разных сессиях, не встречаясь в одном файле. Поэтому части — источник,
витрина — производная: `build.py` собирает её из карточек и вставляет каждый
образец в свой iframe, чтобы части не дрались стилями. Монолит
[foundry-mockups.html](../docs/design/mockups/foundry-mockups.html) (19 экранов)
остаётся разделом «Виды» целиком и режется лениво — только то, что уже
встретилось дважды.

**Отвергнутое хранится наравне с принятым.** Принятое видно на экране,
а отвергнутое не видно нигде — и потому предлагается заново каждой следующей
сессией. В [rejected/](rejected/) — 16 надгробий с причинами: ember, иконка
из роя, тень под кнопкой, световой пролив, градиент по телу, кадрирование роя.
Прежде чем предлагать «а давайте оранжевый акцент» — загляните туда.

**Правило двух.** Один экран не рождает элемент. Часть выделяется, когда
встретилась дважды в разных местах; до этого она — деталь своего экрана.
Иначе система распухает частными случаями и перестаёт быть системой.

## Как добавить элемент из своей сессии

Коротко (полностью — в [parts/README.md](parts/README.md)):

1. Нашли удачное и оно встретилось дважды → пишете один `.html` по контракту
   в папку категории, статус `кандидат`, `lineage` показывает оба места.
2. Крупный кусок целиком (экран, эталон, эскиз) → кладёте как есть
   в `candidates/` + `.md`-карточку рядом. Приводить в порядок не нужно —
   приёмная для того и есть.
3. `python3 design/build.py && python3 design/lint.py` → коммит.

Не спрашивая разрешения: кандидат ничего не ломает, он лежит в своей полосе
и законом не притворяется. Приёмка — отдельный разговор с владельцем.

## Проверка

```
python3 design/build.py    # пересобрать сгенерированное и витрину
python3 design/lint.py     # проверить; exit 1 — есть нарушения
```

Линт держит: ни одного `:root` вне `tokens.css`; ни одного сырого значения
в частях; селекторы части scoped под `.p-<slug>`; карточка на месте и её
токены существуют; сгенерированное совпадает с источником. Правило, которое
не проверяется механически, не существует — это доктрина foundry, и канон уже
доказал её на себе.
