# Часть — контракт

Часть — минимальная единица дизайн-системы: один файл, который открывается
в браузере сам по себе и показывает свой образец во всех состояниях сразу.
Витрина ([../index.html](../index.html)) — не хранилище, а **сборка**: она
собирается скриптом из частей и ничего своего не содержит. Правишь часть —
пересобираешь витрину; правишь витрину руками — теряешь правку.

Этот файл — договор о формате. Он один на все сессии: часть, сделанная в любой
сессии по этому договору, встаёт в систему без переговоров.

## Файл части

Одна часть — один `.html` в папке своей категории. Имя файла = slug:
английский kebab-case, без аббревиатур и усечений (`status-dot`, не `st-dot`).
Русское имя — в карточке.

```html
<!doctype html>
<meta charset="utf-8">
<title>Точка статуса — foundry-desktop</title>
<link rel="stylesheet" href="../../tokens/tokens.css">
<link rel="stylesheet" href="../specimen.css">

<script type="application/json" id="card">
{
  "name": "Точка статуса",
  "slug": "status-dot",
  "status": "принят",
  "height": 120,
  "why": "Показывает состояние change в плотном списке, где нет места словам.",
  "never": "Не применять там, где помещается слово: точка требует легенды, слово — нет. Не применять как единственный носитель смысла — цвет дублируется формой.",
  "tokens": ["sem.success", "sem.warning", "sem.error", "sem.info", "radius.full", "icon.s"],
  "canon": ["06-color.md", "12-icons.md"],
  "swift": null,
  "lineage": "Выделена из foundry-mockups.html §1 и §3 — встретилась дважды."
}
</script>

<style>
  /* Все селекторы части начинаются с .p-<slug> — иначе части передерутся
     стилями в витрине. Проверяется линтом. */
  .p-status-dot { display: flex; gap: var(--space-5); }
</style>

<div class="specimen p-status-dot">
  <!-- Образец: все состояния рядом, чтобы сравнивались глазом,
       а не по памяти (small multiples, Тафти). -->
</div>
```

## Поля карточки

| Поле | Обязательно | Что в нём |
|---|---|---|
| `name` | да | имя по-русски, именительный падеж |
| `slug` | да | английский kebab-case, совпадает с именем файла |
| `status` | да | `принят` · `кандидат` · `отвергнут` |
| `height` | да | высота образца в px для витрины (число, честное) |
| `why` | да | какую пользу приносит; одна фраза без воды |
| `never` | да | **когда НЕ применять** — см. ниже |
| `tokens` | да | какие токены ест; проверяется линтом против `tokens.json` |
| `canon` | да | на какие главы канона опирается |
| `swift` | да | символ реализации или `null` — честное «реализации нет» |
| `lineage` | да | откуда приехала: сессия, PR, номер секции макета |

### Про `never`

Поле дороже всех остальных вместе взятых. `why` пишется легко и всегда — оно
никого ни от чего не останавливает. Система расползается не потому, что кто-то
не знал, зачем нужна кнопка, а потому, что никто не написал, где она неуместна.
Часть без честного `never` в систему не принимается: если границы применения
не нашлись — значит, часть ещё не понята, и это кандидат, а не элемент.

## Категории

Границы — по смыслу, не по «атомам и молекулам»: этот жаргон ничего
не объясняет.

| Папка | Что там | Проверка на принадлежность |
|---|---|---|
| `1-foundation` | палитра, поверхности, текст, отступы, радиусы, шрифт, движение, glow | это значение, а не вещь |
| `2-marks` | иконка, вордмарк, орб, лоадер, DMG | это опознавательный знак продукта |
| `3-elements` | кнопка, поле, чип, бейдж, точка статуса | в одиночку смысла не имеет и про домен не знает |
| `4-blocks` | строка инбокса, карточка канбана, строка диффа, KPI, шаг онбординга | смысл имеет и про домен знает |
| `5-layers` | окно, панель, поповер, модалка, тост, палитра команд, завеса | это контейнер для другого |
| `6-views` | главный экран, ревью, канбан, онбординг, настройки | это целый экран |
| `7-behaviour` | пустое, скелет, ошибка, прогресс, undo, фокус, Reduce Motion | это сквозное поведение, а не вещь на экране |

Сомнение между `3-elements` и `4-blocks` решается одним вопросом: знает ли
часть слова CRISPY, change, стадия? Знает — `4-blocks`.

## Три статуса

**`кандидат`** — не прошёл приёмку. Показан в витрине отдельной полосой,
законом не является: ссылаться на него как на решённое нельзя.

**`принят`** — прошёл приёмку. Закон. Меняется только осознанно и с
обновлением `lineage`.

**Статус — единственная истина о зрелости; папка говорит только о категории.**
Часть может быть оформлена по контракту, лежать в `3-elements` и всё равно
быть кандидатом — так бывает, когда её пришлось сводить из двух разошедшихся
вариантов и сведение требует чьего-то решения. Гонять файл между папками
при смене статуса не надо: витрина разносит части по полосам, читая карточку.

Папка [../candidates/](../candidates/) — **приёмная для неоформленного**: сюда
попадает то, что приехало из сессии крупным куском и ещё не разобрано на части
(целые экраны, эталоны, эскизы). Приёмная не линтуется — в этом и смысл: вход
в систему не должен требовать сначала привести чужую находку в порядок, иначе
находки будут оседать в воркtree и умирать вместе с ними. Так уже было: три
принятых эталона пролежали вне git, потому что им некуда было лечь.

**`отвергнут`** — лежит в [../rejected/](../rejected/) вместе с **причиной
отказа**. Отвергнутое — такой же полноправный житель системы, как принятое,
и стоит оно дороже: принятое видно на экране, а отвергнутое не видно нигде,
и потому предлагается заново каждой следующей сессией. Кладбище существует,
чтобы ember, иконка из роя и тень под кнопкой не воскресали раз в месяц.

## Обряд приёмки

1. Сессия сделала удачное → кладёт файл в `candidates/` со статусом
   `кандидат` и честным `lineage`. Ничего не спрашивает и никого не ждёт.
2. Витрина показывает кандидатов отдельной полосой — их видно сразу.
3. Приёмка: часть переезжает в свою категорию со статусом `принят` — либо
   в `rejected/` со статусом `отвергнут` и вписанной причиной.

**Правило двух.** Один экран не рождает элемент. Часть выделяется в систему,
когда встретилась дважды в разных местах: до этого она — деталь своего экрана,
и обобщать её рано. Правило экономит систему от распухания частными случаями;
`lineage` обязан показывать оба места.

## Что проверяет линт

Правило, которое не проверяется механически, не существует — это доктрина
foundry, и канон уже доказал её на себе: он объявил единственный источник
значений и получил четыре.

- ни одного `:root` вне `tokens.css`;
- ни одного сырого значения (hex, px-отступ, радиус) в частях — только `var(--…)`;
- селекторы части scoped под `.p-<slug>`;
- карточка есть, поля на месте, `status` из трёх допустимых;
- `tokens` карточки существуют в `tokens.json`;
- `canon` указывает на существующие главы;
- `slug` совпадает с именем файла и уникален по системе;
- сгенерированное (`tokens.css`, `Tokens.swift`, таблицы канона) совпадает
  с `tokens.json` — иначе кто-то правил руками.

Запуск — `python3 design/lint.py`, он же на CI.
