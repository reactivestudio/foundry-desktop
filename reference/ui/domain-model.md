# Доменная модель и карта видов

Модель, из которой следуют экраны. Макет — `docs/design/mockups/foundry-mockups.html`.
Значения оформления — только из `docs/design/13-tokens.md`.

## Компоненты продукта

Foundry — не одна программа, а три компонента с разными ролями. Их установка —
часть онбординга (§16).

| Компонент | Роль | Живёт |
|---|---|---|
| **foundry-desktop** | Оркестровка: доски, ревью, очередь. Сам код не пишет. | `/Applications/Foundry.app` |
| **foundry-cli** | Детерминированные операции: worktree'ы, файловая структура, git. Без агента — значит воспроизводимо. Он же MCP-сервер для агента. | `/usr/local/bin/foundry` |
| **foundry-claude** | Знания агента: скиллы, агенты, правила, команды, MCP-регистрация, локальные скрипты. | `~/.claude/` |

**Связи:** desktop → cli (вызывает операции) · cli → агент (MCP-сервер) ·
агент → foundry-claude (читает правила стадий).

**`~/.claude` — папка пользователя.** Туда не устанавливают, а дописывают: своё
кладём с префиксом `foundry-*`, в `CLAUDE.md` дописываем блок в конец, чужое не
трогаем, конфликт имён решаем переименованием своего файла.

## Модель

```
Проект ── лейблы: сервис-нейм · категория (инфра | утилита | бизнес)
└─ Change ── артефакты
   ├─ Роутинг: тип + характеристики → выбор пайплайна
   ├─ Пайплайн (свой у каждого типа)
   │   └─ Стадия: артефакты на входе → артефакты на выходе
   │       └─ часть артефактов → ревью человеком → атрибуция скилла
   └─ атомарные Tasks (появляются на стадии декомпозиции)
```

**Change** — единица работы («задача»). Носит мнемо-имя (Admiral), ветку-хеш,
тип, пайплайн, проект, набор артефактов.

**Артефакт** — не только код. Спека, план, контракт, отчёт, дифф. Каждый артефакт
имеет тип, автора-скилл и статус ревью.

## Роутинг

Change попадает в пайплайн не по алфавиту, а по **типу + характеристикам**.
Решение должно быть объяснимым: видно, какие признаки его породили и как
переопределить вручную.

| Признак | Значения |
|---|---|
| Тип | фича · баг · хотфикс · инцидент · рефакторинг · доки · ресёрч |
| Охват | один сервис · много сервисов |
| Риск | низкий · средний · высокий |
| Категория проекта | инфра · утилита · бизнес |

## Пайплайны (доска на каждый — решение пользователя)

Колонки доски = точные стадии выбранного пайплайна. Пайплайны **разной длины** —
это значимая информация, а не дефект вёрстки.

| Пайплайн | Стадии |
|---|---|
| **Фича · один сервис** | Questions → Research → Design → Structure → Plan → Worktree → Implement → PR (CRISPY 8) |
| **Фича · много сервисов** | + Контракты (после Design), + Интеграция (после Implement) — 10 |
| **Баг** | Repro → Diagnose → Fix → Test → PR (5) |
| **Хотфикс** | Repro → Fix → Ship (3) |
| **Инцидент** | Triage → Mitigate → RCA → Postmortem (4) |

**Ревью — не стадия, а гейт на артефакте.** Оно поперечно пайплайнам: артефакт
на выходе любой стадии может требовать человека. Поэтому у очереди ревью свой
борд с состояниями ревью, а не со стадиями.

## Артефакты по стадиям (пайплайн «Фича · один сервис»)

| Стадия | Вход | Выход | Ревью |
|---|---|---|---|
| Questions | тикет | `questions.md` | — |
| Research | questions.md | `research.md` | — |
| Design | research.md | `design.md` — **спека** | **да** |
| Structure | design.md | `structure.md` | — |
| Plan | structure.md | `plan.md` + **атомарные tasks** | **да** |
| Worktree | plan.md | ветка | — |
| Implement | ветка + tasks | код (дифф) | **да** |
| PR | дифф + вердикт | PR | — |

**Декомпозиция на атомарные tasks происходит на стадии Plan** — до неё второго
уровня не существует, и зум в change показывает только пайплайн.

## Скиллы и обратная связь

Каждый артефакт произведён скиллом (`spec-writer`, `decomposer`, `impl-swift`,
`test-writer`, `router`, `contract-designer`, `researcher`). При ревью человек помечает
**какой скилл отработал неправильно и где** — это фидбек системе на улучшение, а не
просто отказ. Атрибуция видна в диффе поблочно (§12).

**Скилл — тоже артефакт.** Его правило живёт в `foundry-claude/skills/<имя>.md`, и его
ревьюят тем же диффом, что спеку. Это замыкает петлю: §12 родил замечание → §17 превратил
его в правило → следующий прогон его применяет.

**Замечание группируется по причине, а не по времени.** Шесть одинаковых замечаний — это
не шесть случаев, а один пробел в правиле. Повтор и есть диагноз.

| Диагноз | Значок | Что не так | Лечение |
|---|---|---|---|
| Правило есть — не применяет | ▲ | формулировка не срабатывает: ограничение стоит не там, где скилл читает | уточнить правило |
| Правила нет | ○ | случай не покрыт | написать правило |
| Стало правилом | ✓ | правка внесена, возвраты прекратились | ничего, остаётся доказательством |

Алфавит ✓▲○ — тот же, что в чеклисте рисков §3.

**Общего вердикта у скилла нет.** «Одобрить скилл целиком» было бы враньём: у каждого
диагноза своё лечение. И никаких рейтингов — только счёт (`61 артефакт · 14 вернули`) и
исторический базлайн (`до правки — 1 из 3 · после — 1 из 9`).

## Спринт = календарный месяц (решение пользователя)

Граница — календарный месяц. Done схлопывается по границе, сверху переключатель
месяцев. Колонка Done не растёт бесконечно.

## Карта видов

| Вид | Зачем | Уровень | § макета |
|---|---|---|---|
| Инбокс-ревью | что требует решения сейчас | сквозной | §1 |
| Дифф кода | построчное ревью | артефакт | §3 |
| Аналитика | тренды, узкие места | сквозной | §4 |
| **Канбан по change** | вид сверху, доска на пайплайн | change | **§6** |
| **Зум в change** | реальный пайплайн + доска его tasks | change → task | **§11** |
| **Дифф спеки** | прозаический дифф + атрибуция скилла | артефакт | **§12** |
| **Роутинг** | почему этот пайплайн, переопределить | change | **§13** |
| **Сквозной вид по tasks** | застрявшие и загрузка поперёк change | task | **§14** |
| **Канбан ревью артефактов** | очередь ревью: код, спеки, планы | артефакт | **§15** |
| Онбординг | установка с нуля: агент, компоненты, доступ | — | §16 |
| **Скиллы** | куда уходит замечание: диагноз и правка правила | скилл | **§17** |
| **Проект** | лейблы и следующие из них пайплайны | проект | **§18** |
| **Insights · Переделка** | во что обходятся возвраты: токены по стадиям | сквозной | **§19** |

## Измерения: эффективность системы

**§4 и Insights отвечают на разные вопросы.** §4 — «что происходит сейчас»
(пропускная, длительность, где change'и). **Insights — «во что это обходится»**.
Смешивать значит получить дашборд, на который смотрят и ничего не решают.

**Возвраты и токены — не два отчёта, а один: цена возврата = токены, сожжённые
повторно.** По отдельности «14 возвратов» и «1,9М токенов» не значат ничего.

| Метрика | Что это | Почему так |
|---|---|---|
| **Доля переделки** | % расхода, ушедший на повторные прогоны | главная метрика: абсолютный расход сам по себе ни хорош ни плох, «треть работы сделана дважды» — диагноз |
| **Цена возврата по стадии** | токены переделки ÷ число возвратов | показывает, что возвраты не равны: Design ≈ 10К, Implement ≈ 116К |
| **Токены** | факт из сессии агента, отдаёт foundry-cli | точное число |
| **% лимита** | оценка по окну плана, пишется с «≈» | человек в токенах не думает; точность не выдумываем |

**Инверсия наивного чтения.** Возвратов на Design больше, а стоят они в 11 раз
меньше. Значит «меньше возвратов» — неверная цель, верная — **«возвраты раньше»**.
Ранний возврат экономит поздние стадии; поздний — чистый перерасход.

**Детерминированные стадии бесплатны.** Worktree делает `foundry-cli` без агента —
0 токенов. Это аргумент архитектуры: что переносится в cli, то перестаёт стоить.
Ноль (`0`) и отсутствие данных (`—`) выглядят по-разному.

**Срезы раздела** (вкладки): Переделка (§19) · Расход · Скиллы · Пропускная.

### Чего ещё нет (разрыв между моделью и экранами)

| Пропуск | Почему важен |
|---|---|
| **Вход: композер + стадия Questions** | change'и появляются из ниоткуда; Questions — единственная стадия, где агент спрашивает, а человек отвечает: это тред, не дифф |
| **История change'а** | 8 стадий и два возврата — разобрать негде |
| **Закрытие месяца** | спринт календарный, done схлопывается по границе, а обряда закрытия нет |
| **Дифф контракта** | пайплайн «много сервисов» рождает артефакт-схему: не проза и не код; возможно, сведётся к §12 |

## Приёмы под эту модель

- **Семантический зум портфель→элемент без модалок** (Birdview) — change → его tasks.
- **Незыблемые якоря между сценами** (Worldchess) — change держит те же координаты
  при смене уровня, глаз не переискивает контекст.
- **Виджет-сводка = контрол-панель** (Superpedestrian) — переключатель пайплайнов
  одновременно счётчик и легенда.
- **Прогресс как заливка объекта** (Admiral) — стадия закрашивает орб change'а.
- **Трёхзначковая система** (Action-Договоры) — статус ревью артефакта.
- **Цвет только на значимом** (FXPro) — лейблы не радуга: категория формой, не цветом.
